# シーズン4「揺さぶられる」コンセプト

ステータス: 確定(2026-07-05)。執筆ルールは SPEC 10.3、スキーマ拡張は SPEC 3.3・第7章に反映済み

## シーズンの位置づけ

- テーマ: **トレードオフ総合演習**(CONCEPT.md「4. 揺さぶられる」)。**本作の最終シーズン**であり、クリア後にゲーム全体のエンディングへつながる(エンディング自体の仕様は別途)
- 難易度: ★★★★(difficulty: 4)。初見プレイの想定着地は **B**(気を抜くと C。S は周回して狙う)
- ランク閾値は全シーズン共通(S≥280/A≥240/B≥200)。難易度は仕掛けで作る

## 問う力の進化(シーズンの背骨)

| シーズン | 問う力 | プレイヤーの動き |
|---|---|---|
| 1. 要件を聞く | **聞く力** — 曖昧な要望を要件に翻訳する | 文章をよく読んで質問する |
| 2. 形にする | **読む力** — 図と文脈から根拠を見つける | 図を読み、聞くべきことを取捨選択する |
| 3. 支える | **選ぶ力** — 壊れる前提で技術を選定する | ワークロードを裏取りし、障害シナリオと概算で選択肢を絞る |
| 4. 揺さぶられる | **守る力** — 前提が崩れたとき、目的に立ち返って捨てるものを決める | 正しく決めた**後に**揺さぶられ、決め直す |

S1〜S3 は「正しく決めるまで」のゲームだった。S4 は**一度正しく決めた後に前提が変わる**(予算半減・納期前倒し・想定アクセス数の跳ね上がり)ところから本番が始まる。プロジェクトは計画通りに進まない——実務のトレードオフの本当の姿を、最終シーズンで体験させる。

## シーズン4の新しい仕掛け

### 1. 揺さぶりイベント(各話の構造)

エピソード前半は S1〜S3 と同じく「普通に良い判断」をさせる。中盤で**外部イベントが前提を壊す**。

- イベントの瞬間に**パラメータが目に見えて削られる**(予算半減なら budget が大きく低下)。ここから終盤までは「削られた状態からどこまで立て直せるか」のゲームになる → エンジン拡張A(scene への effects)で表現
- **イベント自体は固定**(全ルートで必ず起きる)。序盤の判断は「イベントを回避できるか」ではなく「**被害の大小と立て直しの選択肢**」に効く → エンジン拡張B(next の条件分岐)で、同じ選択でも序盤の flag により展開が変わる形にする。分岐の爆発を防ぎつつ「あのとき◯◯していたおかげで」を体験させる
- 例: 序盤に契約条件(違約金条項)を確認していた者は、納期前倒し要求に対して「動かせない理由」を根拠付きで示せる展開に入れる

### 2. 「何を捨てるか」型の出題

S3 の技術選定(良いものを選ぶ)から一歩進め、**守るものを1つ決め、残りを明示的に諦める**出題にする。

- 罠選択肢の型は「**全部守ろうとする**」(品質も納期も予算も、と言って全部落とす)と「**目的でなく作業量で削る**」(手を付けやすいテスト・ドキュメントだけ削る、一律◯%カット)の2系統を中心に据える
- 正解は常に「案件の目的に立ち返り、守るものを絞り、捨てるものを関係者に見える形で捨てる」。s1e2 で学んだ MoSCoW がここで**制約が動いた後の再仕分け**として回収される(シーズン横断の伏線)
- デブリーフでは「捨てたものはいつ・どう戻すか」(段階リリース・技術的負債の返済計画)まで触れ、「捨てる=失敗」ではないと締める

### 3. 説明・合意形成の配点(trust が主役になる)

何を捨てるか決めた後、「**それを顧客・上司にどう伝えるか**」の選択を各話1問置く。

- 罠の型: 黙って削る(勝手に判断して事後報告)/ 全部持ち帰る(決められず時間だけ溶ける)/ 根性で吸収する(チームに皺寄せして表面上は全部守る)
- 正解の型: トレードオフを構造で見せ、**選択権を相手に渡す**(「AとBは両立しません。どちらを守りますか」)。ただし丸投げとの書き分けに注意——選択肢とそれぞれの帰結・推奨案まで用意して初めて専門家の仕事になる
- 4パラメータのうち trust はこれまで脇役だった。S4 では説明問題の配点を trust に寄せ、「技術判断が正しくても伝え方で炎上する」を体験させる

### 4. 総合演習(S1〜S3 の道具を新規に教えず使わせる)

- ヒアリングによる裏取り(S1)・図から根拠を見つける(S2)・障害シナリオと概算(S3)を、**解法として明示せずに**判断材料の取得手段として要求する。新しい知識の導入は最小限に抑え、難しさは「知識」ではなく「複数制約の同時成立」から作る
- 最終話 s4e3 は**過去シーズンの顧客の案件が成長して戻ってくる**再訪型にする(下記エピソード案)。ゲーム全体の総仕上げであり、クリア後の全体エンディングへの導線になる

## シーズン4固有の執筆原則

「正解のないトレードオフ」というテーマと「S ルート各話1本」の規律の緊張は、S3 の原則A(配点はプロセスに寄せる)よりさらに強く出る。次の3原則で手当てする。

### 原則D: 配点は「捨て方の作法」に置く

- S と A を分けるのは「**どれを守ったか**」ではなく「**捨てるまでに何をしたか**」— (1) 目的・制約の確認(裏取り flag)、(2) 影響の見積り(概算・図読み flag)、(3) 明示的な合意(説明問題の選択)の3手順
- 「何を守るか」自体は、**裏取りで発見できる事実が答えをほぼ強制する案件設定**にする(S3 原則Aの継承)。例: 違約金条項があるから納期だけは動かせない、と聞き出せた者だけが確信を持って他を削れる。前提が緩いまま「あなたならどれを守る?」と聞かない
- ending の最良 variant は `require.flags` + `minTotal`(既存ルール)。flags には上記3手順のプロセス flag を使う

### 原則E: 揺さぶりは1話1回・回復可能に設計する

- 揺さぶりイベントは**各話1回・中盤に固定**。複数回揺さぶると学びが散り、テキストも膨らむ(S3 原則Bの「1話の学びを絞る」の継承)
- イベントの effects で削る量は「最善ルートなら S(≥280)まで立て直せる」範囲に収める。check-routes の「S ルートちょうど1本・最善合計 280〜290」の規律は S4 でも変えない
- 悪いルートは「立て直しの失敗」として C まで落ちてよい(最終シーズンなので、失敗の帰結は S1〜S3 より重く描く。ただし失敗 ending の追体験化・話固有の失敗シーンというフェーズ8の水準を維持)

### 原則F: 揺さぶり前の判断も無駄にしない

- 揺さぶりが来ることをプレイヤーは(1話目以降)メタ的に予期する。それでも**前半の判断が手抜きにならない**ように、前半の flag・パラメータが後半の被害の大小と選択肢の質に効く構造(仕掛け1)を必ず入れる
- 「どうせ前提が変わるなら真面目に設計しても無駄」ではなく「**良い設計は変化への保険になる**」がシーズンを貫くメッセージ。デブリーフで毎回この線に回収する

## エピソード案(3本)

揺さぶりの種類(金・時間・負荷)を1話ずつ変える。

| ID | 仮題 | 揺さぶり | 学び | 図 |
|---|---|---|---|---|
| s4e1 | 予算が半分になりました | 開発中盤、顧客の事業縮小で**予算半減** | 目的に立ち返るスコープの再仕分け(MoSCoW 再訪)。一律カットや「テストだけ削る」の罠。段階リリースという捨て方 | 機能一覧×工数の表+現在の進捗 |
| s4e2 | リリースは1か月早まります | 大型商談を理由に**納期前倒し**。ただし契約に品質基準(受入条件)あり | 品質の何を守るか(落とせる品質と落とせない品質)。クリティカルパスの見極め。「根性で吸収」がチームと品質を壊す帰結 | 工程スケジュール(ガント風)+受入条件の抜粋 |
| s4e3 | フルフルマーケット、再び | s1e2 の EC が成長、TV 露出で**想定アクセス10倍**の見込みが直前に判明。予算・納期・チームの制約が同時に提示される総合問題 | S1〜S3 の全道具の投入。守るものを1つに絞る勇気(セール当日の閲覧と決済だけは落とさない、等)。graceful degradation | 現構成図+実測トラフィック+コスト概算 |

- s4e1・s4e2 の顧客キャラは各話新規(既存シーズンと同様)。アオイ先輩続投
- s4e3 の ending(最良 variant)とデブリーフは、シーズン単体の締めであると同時に**ゲーム全体のエンディングへの導線**として書く(全体エンディング自体は別コンテンツとして S4 実装後に仕様を詰める。TASKS.md 将来参照)
- glossary 候補: スコープ、MVP、段階リリース、クリティカルパス、受入条件、技術的負債、グレースフルデグラデーション、縮退運転 など(1話 4〜7語)
- 商用サービス名を出さない・判断材料はエピソード内で与えきる等、S3 までの共通ルール(SPEC 10〜10.2)はすべて適用

## エンジン拡張(2点。S4 で必要、他は見送り)

### 拡張A: scene ノードに effects(揺さぶりの可視化)

```json
{ "type": "scene", "lines": [...], "effects": { "budget": -25, "customer": -10 }, "next": "..." }
```

- choice の effects と同じ適用規則(clamp・PARAM_KEYS のみ)。UI はシーンテキスト表示後にパラメータ変動を明示する(変動表示の見せ方は choice の feedback カードの流用でよい)
- validateEpisode / smoke-test / check-routes は scene effects を集計に含めるよう拡張

### 拡張B: next の条件分岐(flag による展開の分岐)

```json
"next": [
  { "require": { "flags": ["asked_contract"] }, "to": "shake_mitigated" },
  { "default": true, "to": "shake_full_damage" }
]
```

- scene の `next` と choice option の `next` の両方で、従来の文字列に加えて上記の配列形式を許可する(文字列は後方互換で残す)
- 評価規則は ending variants と完全に同じ(上から順・`require.flags` の全一致・最後は必ず default)— スキーマの一貫性を保ち、エンジンの評価関数も共通化する
- playThrough は state に flags を持つためそのまま動く。check-routes の総当たりは経路が増えるだけで手法は不変

### 見送り(理由付き)

- **choice option の表示条件**(flag がある人だけ選べる選択肢): 見えている選択肢の数が可変になると、正解位置分散・文長監査・総当たり検証の前提が崩れる。拡張Bの「同じ選択でも帰結が違う」で教育目的は満たせる
- **debrief の variants 化**: T32 で不採用済み。ending variants の順序評価(default 到達≒キー flag 未取得)で回収する既存手法を維持。S4 の脚本で ending 側に収まらない場面が実際に出たら再検討

## 見せ方・難易度の調整

- choice は 4〜5 回、無傷の正解なし、S ルート各話1本、正解位置の分散・文長・極端語ルール(SPEC 10〜10.1)はすべて維持。smoke-test の first/last/longest 戦略 assert がそのまま効く
- 揺さぶりイベントの scene は演出上の山場。立ち絵の表情・地の文で「空気が変わった」ことを描く(テキスト量はフェーズ8で増強した ending 水準を目安に)
- 設問パターンの使い回し禁止(SPEC 10)に特に注意: 3話とも「揺さぶり→再仕分け→説明」の骨格を共有するため、**出題の切り口・罠の中身・説明問題の状況を話ごとに明確に書き分ける**(s4e1 は削る対象の選定、s4e2 は品質基準との衝突、s4e3 は複数制約の同時成立、と主役を変える)
- リザルトの周回促しコピーは S4 文脈に合わせる(例: 「別のものを守った世界線も見てみよう。何を捨てたかで、燃え方が変わる」)
