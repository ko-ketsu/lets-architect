# シーズン2「形にする」コンセプト

ステータス: 確定・実装済み(2026-07)

## 背景(MVP のプレイヤーフィードバック)

MVP を実ユーザーに使ってもらった結果:

- 方向性・形式はおおむね好評
- ただし**シーズン1はほとんどのユーザーが一発で S** を取れてしまい、「簡単すぎる」と感じられている

判断: シーズン1は導入なので**易しいまま据え置く**(難しすぎて離脱されるより「いけるぞ」という成功体験を優先)。要領を得たユーザー向けに、**シーズン2からそこそこ頭を使う難易度**に引き上げる。

## 難易度カーブ(シーズン別の設計目標)

| シーズン | 位置づけ | 初見プレイの想定着地 |
|---|---|---|
| 1. 要件を聞く | 導入。操作と世界観に慣れ、成功体験を得る | **S**(一発クリアで OK)※下記の微調整あり |
| 2. 形にする | 応用。図を読み、文脈で判断する | **A〜B**(S は狙って周回して取る) |
| 3. 支える | 発展。複数の制約を同時に扱う | B(S2 の仕掛け+制約の同時成立) |
| 4. 揺さぶられる | 総合演習。正解のないトレードオフ | B〜C も普通(周回前提) |

ランク閾値(S≥280/A≥240/B≥200)は**全シーズン共通のまま**。難易度は閾値ではなく、選択肢と effects の設計で作る(下記)。閾値をシーズンごとに変えると「同じ S でも価値が違う」ことになり、比較の楽しみが壊れるため。

### シーズン1への微調整

シーズン1は易しいまま据え置くが、**各話の最終問題(3問目)だけは、気を抜いて答えると間違える構成に改稿する**:

- それまでの状況(ヒアリングで得た情報・顧客の事情)をちゃんと踏まえていないと正解を選べない選択肢構成にする
- S ルートは各話1本のまま維持。気を抜いた回答は A 落ち程度のペナルティ(即 B/C まで落とさない)
- 対象: s1e1 / s1e2 / s1e3 の最終 choice ノードの選択肢文面と effects

## シーズン2で「頭を使わせる」5つの仕掛け

エンジン変更がほぼ不要な順。すべて既存スキーマ(effects / flags / next 分岐 / ending variants)+ SPEC 第8・9章の image/portrait 拡張の範囲で実現できる。

### 1. 正解の根拠を「図」に置く(図解出題の使い方)

本文のテキストだけ読むと選択肢が同格に見えるが、**添付の図(ER図・シーケンス図・構成図)を読むと根拠が見つかる**構造にする。シーズン1の「文章をよく読めば分かる」から「図と突き合わせないと分からない」へ。

例: 「このテーブル設計で注文履歴の要望を満たせるか?」— prompt だけでは判断不能。ER 図を見ると注文と商品が直接多対多で結ばれており、価格改定時に過去の注文金額が変わってしまうことに気づけるか。

### 2. 選択肢の「もっともらしさ」を均質にする

シーズン1の反省: 正解の選択肢が文面からして一番丁寧に見えてしまう。シーズン2では**全選択肢を「別の文脈でなら正解」にする**。

- 罠選択肢 = 誤りではなく「この案件の前提では裏目に出る正論」(例: 「将来に備えて汎用的に設計する」は正論だが、この案件は納期3週間の PoC)
- 判別の手がかりは、エピソード序盤のヒアリングや図の中に置く
- デブリーフで「どういう案件ならその選択が正解だったか」まで解説し、文脈依存の判断力として回収する

### 3. ヒアリングの取捨選択(全部は聞けない)

問題構成は変えない(シーズン1と同じ「ヒアリング → 設計判断」の流れ)。変わるのはヒアリングパートの性質だけで、**質問がアーキテクチャを決めるのではなく、質問の選び方が「後の設計判断を根拠を持って選べるか、当て推量になるか」を決める**。

- シーズン1: 質問は聞けば必ず加点(選ぶ意味が薄かった)
- シーズン2: 作中の時間制約として提示(例: 「打ち合わせは残り15分。聞けるのはあと2つだ」)。質問候補4つから2つを選ぶ
- 質問の答えはポイントではなく**情報**として返る(内部的には flag に記録)
- その後の設計判断の選択肢は、何を聞いたかに関係なく**同じものが出る**。正しい質問をしていれば根拠を持って選べるが、聞き逃していると選択肢がどれも同格に見える
- 外れ質問(答えを聞いても後の判断に効かない)を混ぜる

具体例(s2e1 想定): 質問候補「①過去の注文の金額表示はどうしたい?」「②商品の価格って変わります?」「③繁忙期の注文件数は?」「④画面の配色の好みは?」から2つ。②を聞くと「セールで頻繁に価格改定する」と判明し、後半の設計判断「注文明細に価格をコピーして持つ vs 商品マスタを参照する」で確信を持って選べる。④は外れ。聞き逃した場合はデブリーフで「あの時◯◯を聞けていれば」と回収し、理不尽感ではなくリプレイ動機に変える。

実装はノード分岐+flags で表現できる(質問1回ごとに choice ノードを経由し、聞いた質問を flag に記録)。

### 4. 序盤の選択が中盤の状況を変える(本物の分岐)

シーズン1はほぼ一本道+feedback だった。シーズン2では option の `next` を本格的に使い、**序盤の判断で中盤のシーンが分岐**する。悪い分岐も即詰みにはせず、リカバリー可能だがコスト(パラメータ)を払う。最良 ending variant は `minTotal` と `require.flags` の**両方**を要求し、「点が高いだけ」でも「気づきだけ」でも S 相当の結末に届かないようにする。

### 5. 無傷の正解を置かない(点数の締め付け)

- 全選択肢が**どこかの軸にマイナスを含む**(トレードオフの体感)。シーズン1のような +10/+5 のノーリスク択を置かない
- choice 回数を 3 回 → **4〜5 回**に増やす(ルート空間が 27 → 100 超に広がり、総当たり的な一発 S がほぼ不可能になる)
- S ルートは全ルート中 **1 本だけ**を維持(brute-force スクリプトで全ルート合計を検証してから確定する。シーズン1と同じ手順)

## エピソード案(3本)

テーマは CONCEPT.md 通り「アプリ設計・API設計」。各話に図解 1〜2 枚。

| ID | 仮題 | 学び | 図 |
|---|---|---|---|
| s2e1 | その画面、テーブルが泣いてます | 画面起点 vs データ起点。履歴データの持ち方、多対多の交差エンティティ | **ER図**(顧客の要望通りに作った場合の破綻を読み取る) |
| s2e2 | 使いにくいAPIはこうして生まれる | API の粒度・命名・チャッティな呼び出し(N+1)。利用者視点の設計 | **シーケンス図**(画面表示に API 呼び出しが何往復必要かを数える) |
| s2e3 | その変更、壊れるのはどこ? | 既存 API の変更影響。後方互換・バージョニング・段階的移行 | **構成図/依存図**(どのクライアントが古いフィールドに依存しているか探す) |

顧客キャラは各話新規(シーズン1と同様)。アオイ先輩は続投し、シーズン2から立ち絵付き(SPEC 第9章)。

## 見せ方の調整(難易度を「仕様」として伝える)

- エピソード選択画面に**難易度表示**を追加(★☆☆ / ★★☆ / ★★★)。`data/index.json` に `difficulty: 1〜3` フィールドを足す
  - これによりシーズン1の易しさが「手抜き」ではなく「導入」として伝わる
- リザルト画面の S 未達時コピーで周回を促す(例: 「S ルートはこの話に1本だけ。ヒアリングの選び方から見直してみよう」)
