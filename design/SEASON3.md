# シーズン3「支える」コンセプト

ステータス: ドラフト(レビュー中)

## シーズンの位置づけ

- テーマ: **インフラ・アーキテクチャ**(CONCEPT.md「3. 支える」)
- 難易度: ★★★(difficulty: 3)。初見プレイの想定着地は **B**(S は周回して狙う)
- ランク閾値は全シーズン共通(S≥280/A≥240/B≥200)。難易度は仕掛けで作る

## 問う力の進化(シーズンの背骨)

| シーズン | 問う力 | プレイヤーの動き |
|---|---|---|
| 1. 要件を聞く | **聞く力** — 曖昧な要望を要件に翻訳する | 文章をよく読んで質問する |
| 2. 形にする | **読む力** — 図と文脈から根拠を見つける | 図を読み、聞くべきことを取捨選択する |
| 3. 支える | **選ぶ力** — 壊れる前提で技術を選定する | ワークロードを裏取りし、障害シナリオと概算で選択肢を絞る |

シーズン2は「与えられた設計の問題点を図から見抜く」だった。シーズン3は**自分で構成要素を選定する**側に回る。API ゲートウェイ・キャッシュ、VM/コンテナ/サーバーレス、SQL/NoSQL——どれも実務で使われる「正解候補」ばかりの中から、この案件の前提で裏目に出ないものを選ぶ。

## シーズン3の新しい仕掛け

### 1. 技術選定型の出題(全選択肢が「どこかでは正解」)

シーズン2の罠原則「別の文脈なら正解」を、選択肢の一部ではなく**出題全体の構造**にする。VM か、コンテナか、サーバーレスか——技術選定にはそもそも万能の正解がなく、ワークロード・チーム・制約との相性で決まる。

- 判別の手がかりは**ワークロードの裏取り**(仕掛け3)と**障害シナリオの想定**(仕掛け2)に置く。裏取りせずに選ぶと、どの選択肢ももっともらしく見える
- 罠選択肢の型は「**銀の弾丸信仰**」を中心に据える: 「とりあえず Kubernetes」「モダンだから NoSQL」「サーバーレスなら運用不要」——流行・イメージで選ぶと、この案件の前提(チームの運用力・アクセスパターン・整合性要求)で裏目に出る
- デブリーフで「どういう案件ならその選択が正解だったか」を必ず回収する(SPEC 10.1 の罠原則と同じ)
- **チームの運用力も選定基準**であることを学びに含める(例: 2人のチームに k8s クラスタは運用できない)。技術の優劣ではなく相性で選ぶ、がシーズンを貫くメッセージ

### 2. 「壊れたら?」の思考実験(障害耐性を選定根拠にする)

障害対応そのものではなく、**選定の段階で障害シナリオを想定できるか**を問う。

- 出題例: 「この構成でキャッシュサーバーが落ちたら何が起きる?」— 構成図を読み、リクエストが素通りして DB に直撃する(キャッシュ雪崩)ことに気づけるか
- 「単一障害点はどこか」「フェイルオーバーに何秒かかるか、その間ユーザーには何が見えるか」を選択肢の判断材料にする
- 出題図のネタバレ防止ルール(SPEC 10.1)を厳守: 図には**現状の構成という事実のみ**を描く。単一障害点の強調・障害時の挙動の注釈は描かない(気づくこと自体が問題)

### 3. ワークロードの裏取り(ヒアリングの発展形)

シーズン2の「質問は2つまで」の対象を、顧客の要望から**システムとチームの特性**に発展させる。

- 質問候補の例: 「読み書きの比率は?」「ピークは予測できる形?(セール型/常時なだらか型)」「整合性が一瞬崩れると業務はどうなる?」「運用チームは何人で、何の経験がある?」
- 答えは flag として返り、後の選定判断の根拠になる(仕組みはシーズン2と同じ: 4候補から2つ、外れ質問を混ぜる)
- 聞き逃すと選定の選択肢がどれも同格に見える——シーズン2で確立した構造をそのまま活かす

### 4. 概算で裏を取る(補助スキル)

選定の根拠を数字で確かめる場面を各話に1問程度織り込む。

- 例: 「ピーク 2,000 req/s、キャッシュヒット率 90% なら DB に届くのは?」「可用性 99.9% のコンポーネントを直列に3つ繋ぐと全体は?」(可用性は掛け算で下がる)
- 計算は**桁と大小関係が分かれば解ける**レベルに抑える(電卓が要る精度は求めない)。正確な式はデブリーフで解説
- 図に描くのは個々の実測値・仕様値のみ。合計・掛け算の結果は描かない(SPEC 10.1 の数値版)

### 5. 複数制約の同時成立(最終話で総合)

- 予算・SLA・チームの運用力など**複数の制約が同時に提示**され、1つの軸だけ最適化すると他が破綻する選択肢構成
- 序盤の選定が中盤の前提を変える(シーズン2の仕掛け4の深化): 序盤の選定次第で中盤に顕在化する問題が変わり、リカバリーのコストを払う
- 最良 ending variant は `require.flags`(正しい裏取り・障害シナリオへの気づき)+ `minTotal` の両方(シーズン2と同じ)

## エピソード案(3本)

レイヤーごとに1話ずつ、選定対象を変える。

| ID | 仮題 | 選定対象 | 学び | 図 |
|---|---|---|---|---|
| s3e1 | その入口、守りはあるか | **ネットワーク構成**: API ゲートウェイ・キャッシュ・CDN の要否と置き場所 | キャッシュはどこに置くか(CDN/アプリ前段/アプリ内)。キャッシュが落ちたら何が起きるか(雪崩)。ゲートウェイの役割(認証・レートリミットの集約) | 構成図+トラフィック実測値 |
| s3e2 | VM でいいじゃん、は本当か | **実行基盤**: VM / コンテナ / サーバーレス | ワークロードの形(バースト型か常時型か)とチームの運用力で選ぶ。コールドスタート・実行時間制限などの制約。「モダン=正解」ではない | ワークロードのグラフ(時間帯別リクエスト数)+候補比較の構成図 |
| s3e3 | そのデータ、失えますか | **データストア**: SQL / NoSQL、冗長構成 | 整合性 vs 可用性のトレードオフ。単一障害点の見つけ方。SLA からの逆算(可用性の掛け算)。総合問題(仕掛け5) | 構成図+各コンポーネントの可用性実績・データ特性の表 |

- 顧客キャラは各話新規(既存シーズンと同様)。アオイ先輩続投・立ち絵付き
- glossary 候補: API ゲートウェイ、CDN、キャッシュヒット率、コールドスタート、コンテナ、サーバーレス、単一障害点、フェイルオーバー、結果整合性、レートリミット など(1話 4〜7語)
- 固有の商用サービス名(AWS Lambda 等)は本文では使わず一般名詞で書く(陳腐化と偏りを避ける)。デブリーフの補足で「実務では◯◯などが該当」と触れる程度は可

## 見せ方・難易度の調整

- choice は 4〜5 回、無傷の正解なし、S ルート各話1本(シーズン2と同じ規律。SPEC 10.1)
- 正解位置は設問ごとにばらす(SPEC 10。smoke-test が自動検知)
- 数値表・比較図は横長になりがち → 既存の拡大モーダル+`overflow-x: auto` で対応(SPEC 8.1 のまま)
- リザルトの周回促しコピーはシーズン3の文脈に合わせる(例: 「別の基盤を選んだ世界線も見てみよう。裏目の出方が変わるはず」)

## 実装への影響まとめ

| 項目 | 変更 |
|---|---|
| エンジン | **変更なし**。仕掛け1〜5 はすべて既存スキーマ(image / flags / next 分岐 / ending variants)で表現できる |
| index.json | シーズン3とエピソード3本を追加(difficulty: 3) |
| UI | 変更なし |
| smoke-test / check-routes | 変更なし(既存の assert がそのまま効く) |
| データ | s3e1〜s3e3 の JSON、diagrams/*.svg(各話1〜2枚、数値表・グラフ含む)、顧客キャラ立ち絵3体 |

## レビューで確認したい点

1. **3話のレイヤー分担**(ネットワーク → 実行基盤 → データストア)— この切り方・順序でよいか
2. **「壊れたら?」の思考実験** — 障害耐性の問い方としてこの形(選定根拠としての障害シナリオ想定)でイメージに合っているか
3. **概算の扱い** — 補助スキルとして各話1問程度に抑える配分でよいか
4. **商用サービス名を出さない方針** — 一般名詞(コンテナ基盤・マネージドDB等)で通してよいか
